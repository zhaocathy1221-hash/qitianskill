# 评测用例

分别从事实准确、活人感、因果清楚、来源覆盖和结构质量五个维度打 0—2 分。总分达到 8/10，且任何维度不得为 0，才算通过。

## 用例一：创业者长篇访谈

原始素材包含多个项目、运营建议、一次创业失败、后来的团队经历和带有反思的结尾。

预期表现：

- 在人物线、项目线和团队线中确定一条主线；
- 保留失败和判断形成过程，而不只是结论；
- 当后半段操作内容削弱人物时，将其移入支线文章；
- 保留有原文依据的情绪变化；
- 区分营业收入、销售额和利润。

## 用例二：自由生活与连续的微小成果

原始素材包含令人向往的当下生活、敏感家庭经历、漫长的离职决定、几个失败或上限不高的项目、微小正反馈，以及后来表现较好的电商项目。

预期表现：

- 将补足读者心力识别为重要结果；
- 保留“失败 → 微小成果 → 恢复信心 → 后来成功”的链条；
- 至少保留三个有来源依据的具体场景；
- 将敏感人生经历标记为待确认；
- 区分两小时高度专注工作与四至五小时总工作时间；
- 区分峰值收入与稳定收入；
- 如果运营方法淹没人物，建议拆分人物篇和项目篇。

## 用例三：虚构情绪回归测试

逐字稿包含事实和方法，但嘉宾没有说明自己的感受。

预期表现：

- 不得擅自增加“我很焦虑／兴奋／崩溃”等第一人称表述；
- 描述可观察的事件，或把情绪列为待确认；
- 不得虚构一条逆袭或自我救赎路径。

## 用例四：局部修改请求

用户已经确认大纲，随后要求加强某一章的情绪。

预期表现：

- 只修改受影响的章节；
- 补回有来源依据的场景卡片；
- 保持无关章节稳定；
- 说明发生变化的来源编号。

## 发布检查清单

- 校验 `SKILL.md`；
- 运行用例一、二、三；
- 确认 Skill 中没有私人逐字稿、私人链接、访问令牌或个人身份信息；
- 确认回答长度只是默认建议，不会导致机械截断；
- 确认内容改稿 Skill 仍然保持独立。
