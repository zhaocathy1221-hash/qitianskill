# 从录音到访谈稿：使用说明书

这份说明适合手里已经有访谈录音、飞书妙记链接、飞书文档逐字稿，想用 `longform-interview-editor` 生成一篇有活人感的长篇人物访谈的人。

## 它能解决什么卡点

很多长访谈不是卡在“不会写”，而是卡在中间流程太乱：

1. **录音材料散**：录音、妙记、飞书文档、聊天记录混在一起，不知道从哪里开始。
2. **逐字稿太长**：几十分钟到几小时的材料，一打开就想放弃，很容易只抓到观点，漏掉人物变化。
3. **容易写成方法论**：项目细节很多，但人物为什么选择、为什么还能继续、哪里受挫、哪里获得信心，被写没了。
4. **事实和情绪没依据**：成稿里出现“焦虑、崩溃、坚定、释然”等词，但原文没有支撑，容易失真。
5. **数字口径容易混**：GMV、营收、利润、到账、订单量、峰值、稳定值经常被写混。
6. **改稿难追溯**：用户说“这里不够有活人感”，但编辑不知道该补哪段原文、删了哪些高价值素材。

这个流程把访谈整理拆成 6 个稳定步骤：

`录音/妙记 -> 逐字稿 -> 来源台账 -> 人物变化链条 -> 主题大纲 -> 问答稿`

这样做的好处是：先把证据和人物线搭稳，再进入写作，而不是直接从长逐字稿跳到成稿。

## 大概能提效多少

以下是经验型预估，不是固定承诺，实际取决于录音长度、逐字稿质量和成稿要求。

| 环节 | 传统做法 | 使用这个流程后 | 主要节省 |
|-|-|-|-|
| 找素材 | 反复听录音、翻逐字稿 | 先建立来源台账和编号 | 少翻原文，减少遗漏 |
| 搭结构 | 边读边想大纲，容易反复推倒 | 先提人物变化链条，再定主线 | 结构返工减少 |
| 写初稿 | 直接从逐字稿改写，容易写散 | 按大纲和来源卡片重构问答 | 初稿更快成形 |
| 查事实 | 成稿后逐段回查 | 数字、原话、情绪在台账里先标记 | 降低事实错误 |
| 局部修改 | 牵一发动全身 | 按来源编号补丁式修改 | 改稿更可控 |

一般情况下：

- **30-60 分钟访谈**：从素材到可读初稿，通常可节省约 `40%-60%` 的整理时间。
- **1-2 小时访谈**：在素材复杂、项目线很多时，通常可节省约 `50%-70%` 的结构梳理和回查时间。
- **多轮改稿**：因为有来源台账，局部补细节、补情绪、删敏感内容时，返工成本会明显下降。

更重要的不是单纯变快，而是减少三类质量损失：漏掉关键小正反馈、把人物写成方法论、把无依据情绪写进嘉宾口中。

## 你最终会得到什么

建议完整产出四份文件：

1. `transcript.md`：原始逐字稿，保留说话人和时间戳。
2. `source-ledger.md`：来源台账，记录事实、数字、原话、场景、情绪和敏感信息。
3. `outline.md`：人物变化链条和主题大纲。
4. `interview-draft.md`：完整访谈问答稿。

如果时间紧，也可以只要 `outline.md` 或 `interview-draft.md`。

## 第 0 步：准备环境

### 安装 Skill

把 `longform-interview-editor/` 放到 Codex skills 目录：

```bash
mkdir -p ~/.codex/skills
cp -R longform-interview-editor ~/.codex/skills/
```

然后重启 Codex 或刷新 skill 列表。

### 如果要处理飞书素材

需要本地可用 `lark-cli`，并且已经完成飞书登录授权。

常见检查命令：

```bash
lark-cli --version
lark-cli auth status
```

如果提示没有登录，按 CLI 提示执行登录。不同团队权限配置不同，遇到缺少 scope 时，按错误提示补授权即可。

## 第 1 步：判断你手里的素材类型

按下面三种情况选择入口。

### 情况 A：你已经有飞书妙记链接

飞书妙记链接通常长这样：

```text
https://example.feishu.cn/minutes/obcnxxxxxxxxxxxxxxxxxxxx
```

从链接最后提取 `minute_token`：

```text
obcnxxxxxxxxxxxxxxxxxxxx
```

然后获取逐字稿：

```bash
lark-cli vc +notes --minute-tokens obcnxxxxxxxxxxxxxxxxxxxx --output-dir ./materials
```

命令完成后，通常会在 `./materials/` 下生成包含 `transcript.txt` 的目录。把它复制或重命名成：

```text
transcript.md
```

如果你还想下载原始音视频备份：

```bash
lark-cli minutes +download --minute-tokens obcnxxxxxxxxxxxxxxxxxxxx --output-dir ./materials
```

### 情况 B：你只有本地录音或视频文件

支持常见音频和视频格式，例如 `mp3`、`m4a`、`wav`、`mp4`、`mov`。

限制：

- 时长不超过 6 小时；
- 文件不超过 6 GB。

先上传到飞书云空间，得到 `file_token`：

```bash
lark-cli drive +upload --file ./recording.m4a
```

从返回结果里找到 `file_token`，然后生成飞书妙记：

```bash
lark-cli minutes +upload --file-token FILE_TOKEN
```

命令会返回一个 `minute_url`，从里面提取 `minute_token`：

```text
https://example.feishu.cn/minutes/obcnxxxxxxxxxxxxxxxxxxxx
```

继续获取逐字稿：

```bash
lark-cli vc +notes --minute-tokens obcnxxxxxxxxxxxxxxxxxxxx --output-dir ./materials
```

如果刚上传完就获取不到逐字稿，通常是妙记还在转写。等几分钟后重试。

### 情况 C：你已经有飞书文档或本地逐字稿

如果已经有本地逐字稿文件，直接整理成 `transcript.md`。

最低要求：

- 保留说话人；
- 尽量保留时间戳；
- 不要在原始稿里先大量改写；
- 不确定的识别错误先标记，不要凭空修正。

如果逐字稿在飞书文档里，可以先导出 Markdown 或让 Codex 读取文档内容，再保存为 `transcript.md`。

## 第 2 步：清理逐字稿

目标不是润色，而是让原始材料可读、可追溯。

你可以这样请求 Codex：

```text
使用 $longform-interview-editor，清理这份逐字稿。

要求：
1. 保留说话人和时间戳。
2. 删除无意义口头禅和明显识别噪音。
3. 保留有意义的口述特征、自我修正、不确定感和笑声。
4. 无法确认的内容标记为【待确认】。
5. 不要改写成正式文章。
```

产出保存为：

```text
transcript-clean.md
```

## 第 3 步：建立来源台账

这一步决定后面的稿子是否可信。不要跳过。

请求示例：

```text
使用 $longform-interview-editor，基于 transcript-clean.md 建立来源台账。

请按完整事件和说话轮次编号，不要按固定字数切割。
重点提取：
- 事实与数字；
- 原话与口述特征；
- 具体生活场景；
- 欲望、目标与现实压力；
- 决策过程；
- 失败、代价、怀疑与矛盾；
- 微小正反馈；
- 后来可见的成果；
- 可以移入支线文章的方法内容；
- 需要确认的敏感信息。
```

产出保存为：

```text
source-ledger.md
```

## 第 4 步：提炼人物变化链条

请求示例：

```text
使用 $longform-interview-editor，基于 source-ledger.md 提炼人物变化链条。

请按这个结构输出：
想要的生活 -> 起点限制 -> 第一次尝试 -> 失败或代价 -> 微小正反馈 -> 获得的能力或信心 -> 再次尝试 -> 可见成果 -> 成为现在的自己。

每一环都标注来源编号。素材不足的地方标记【待确认】。
```

这一版不用追求文采，重点是把因果链条理清楚。

## 第 5 步：确定主线和大纲

先让 Codex 判断主线，不要直接写全文。

请求示例：

```text
使用 $longform-interview-editor，基于 source-ledger.md 和人物变化链条，产出主题大纲。

请先判断这篇更适合：
- 人物线；
- 项目线；
- 团队线。

如果人物经历和项目方法都足够丰富，请建议是否拆成两篇。

每章请包含：
1. 章节标题；
2. 本章作用；
3. 推进的变化或问题；
4. 来源编号；
5. 关键场景或原话；
6. 由事件支撑的情绪；
7. 敏感信息或待补信息；
8. 应移入支线文章的素材。
```

产出保存为：

```text
outline.md
```

## 第 6 步：生成完整访谈稿

确认大纲方向后，再写问答稿。

请求示例：

```text
使用 $longform-interview-editor，基于 transcript-clean.md、source-ledger.md 和 outline.md，重构一篇完整人物访谈问答稿。

重点：
- 要有活人感，不要只总结方法；
- 写出嘉宾想要什么、经历过什么失败、哪些微小正反馈让他继续；
- 所有事实、数字、原话和情绪都要能对应来源；
- 问题可以由编辑重构，但不能虚构嘉宾没有回答过的内容；
- 单轮回答默认 150-300 字，完整场景可放宽到 300-500 字；
- 项目方法太多时移入支线，不要淹没人物。

输出：
1. 访谈正文；
2. 审查记录：素材覆盖、删除内容、待确认事实、数字口径、编辑重构问题。
```

产出保存为：

```text
interview-draft.md
```

## 第 7 步：机械检查和人工审查

如果稿件是 Markdown，可以运行：

```bash
python3 longform-interview-editor/scripts/validate_interview_draft.py interview-draft.md
```

这个脚本会检查：

- 章节编号是否连续；
- 是否有重复章节标题；
- 问答段落是否过短或过长；
- 是否缺少加粗说话人标签。

脚本只是机械检查，不等于内容质量判断。

人工还要重点检查：

- 数字有没有混淆：GMV、营收、利润、到账、订单量；
- 峰值和稳定值有没有混淆；
- 情绪是否有原文依据；
- 敏感经历是否需要确认；
- 有没有把编辑推断写成嘉宾第一人称；
- 方法内容是否淹没人物主线；
- 有没有删掉“为什么这个人还能继续”的关键小正反馈。

## 常见问题

### 1. 没有飞书，只有录音，可以用吗？

可以。你需要先用任何可靠转写工具得到逐字稿，再从第 2 步开始。

这个 skill 的核心不是转写，而是从逐字稿到访谈稿的重构。

### 2. 一定要建立来源台账吗？

强烈建议要。

没有来源台账，长访谈很容易出现三种问题：

- 悄悄删掉关键小正反馈；
- 把项目方法写得太多，人物线断掉；
- 虚构或夸大人物情绪。

### 3. 能不能直接把逐字稿变成全文？

可以，但质量通常不如分步做。

如果时间很紧，可以这样请求：

```text
使用 $longform-interview-editor，把这份逐字稿直接整理成一篇人物访谈。
请先在内部完成来源台账和人物变化链条，再输出正文和审查记录。
```

### 4. 为什么要保留失败和小项目？

因为长访谈的心力往往来自这里。

一个最终没做大的项目，可能提供了现金流、信心、产品感、平台经验或再次尝试的勇气。如果直接删掉，读者会看不懂这个人为什么能走到后来。

### 5. 什么时候要拆成两篇？

当素材里同时有很强的人物经历和很完整的方法教程时，建议拆：

- 人物篇：选择、挫折、变化、心态、生活方式；
- 项目篇：机会判断、执行步骤、SOP、工具、数据、坑。

不要让教程内容淹没人物，也不要为了人物故事牺牲可复用的方法内容。
